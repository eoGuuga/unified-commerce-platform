# Auditoria Completa — GTSoftHub/UCM (2026-05-20)

## 1. Resumo Executivo

O GTSoftHub é uma plataforma de comércio unificado (WhatsApp + PDV + Loja Online) multi-tenant, já em produção com 1 cliente ativo (Loucas por Brigadeiro). A auditoria cobriu backend, frontend, infraestrutura, segurança e documentação.

**Nota geral: 6.5/10** — Acima da média para solo dev com produto em produção. Bases sólidas (RLS, testes, backup), mas com gaps operacionais e dívida técnica concentrada.

## 2. Backend (Nota: 7/10)

### Pontos fortes
- Arquitetura NestJS com módulos bem separados e DI correto
- Multi-tenancy via RLS com `SET LOCAL` (implementação correta)
- 49 arquivos de teste, ~653 testes passando
- Segurança: JWT + Helmet + Throttler + ValidationPipe
- Idempotência em POSTs críticos (SHA-256 + Redis + Postgres)
- LLM Router recém-implementado (direção certa)

### Problemas críticos
- `whatsapp.service.ts` com 14.845 linhas — God Object, risco de regressão em qualquer mudança
- CSRF desabilitado em produção (`CSRF_ENABLED=false`)
- Comparação de CSRF token vulnerável a timing attack (usa `!==` ao invés de `crypto.timingSafeEqual`)
- `CacheService.deletePattern` usa comando `KEYS` (bloqueia Redis em datasets grandes)
- Extração de tenant via header em rotas webhook pode permitir spoofing em produção
- `uncaughtException` handler loga mas não encerra processo (pode continuar em estado corrompido)

### Métricas
- 141 arquivos .ts de produção, ~31.600 LOC
- 49 arquivos de teste, ~12.000 LOC
- 15 entidades, 10 módulos, 30 services
- 25 usos de `any` em código de produção

## 3. Frontend (Nota: 6/10)

### Pontos fortes
- Next.js 16 + React 19 + Tailwind 4 (stack moderna)
- shadcn/ui + Radix (acessibilidade built-in)
- TypeScript strict: true, apenas 30 `any`
- 21 arquivos de teste (vitest + testing-library)
- Error boundaries por rota
- Decomposição parcial da loja (10 componentes extraídos)

### Problemas críticos
- `pdv/page.tsx` (1791 linhas, 18 `any`), `loja/page.tsx` (1925 linhas), `pedido/page.tsx` (1351 linhas) — monolitos client-side
- Todas as páginas principais são `'use client'` — zero SSR para dados
- Duas bibliotecas de toast (react-hot-toast + sonner) — inconsistência
- SWR e Supabase instalados mas não usados (deps mortas)
- Sem testes para páginas principais (só utils e componentes extraídos)

### Métricas
- ~100 arquivos fonte
- 24 páginas/layouts no app/
- 80+ componentes
- 21 arquivos de teste

## 4. Infraestrutura (Nota: 6.5/10)

### Pontos fortes
- Docker Compose com resource limits em produção
- SSL com TLS 1.2/1.3 + HSTS
- Rate limiting em dupla camada (Nginx + NestJS)
- Firewall UFW (só 22/80/443)
- Backup diário + offsite B2 criptografado (30 dias)
- Drill de restore mensal automatizado
- DR plan documentado (RPO 24h, RTO 2h)

### Problemas críticos
- **Zero monitoramento** — sem Prometheus, sem uptime monitor, sem alertas de crash
- **Deploy manual via SSH** — sem CD automatizado
- **RPO de 24h** para plataforma de comércio (pode perder 1 dia de pedidos/pagamentos)
- **Containers rodam como root** (sem `user:` no compose)
- **Sem CDN** — todo tráfego estático passa pelo nginx da VPS
- **Dois ambientes na mesma VPS** sem isolamento claro de recursos

### Topologia
- VPS única (OVH) rodando prod + test
- 5 containers prod: postgres, redis, backend, frontend, nginx
- 6 containers test: postgres, redis, backend, frontend, nginx, evolution
- Nginx expõe 80/443, roteia para ambos os stacks

## 5. Segurança (Nota: 6.5/10)

### Implementado corretamente
- RLS multi-tenant com session variable
- JWT com validação de secret mínimo (32 chars)
- Helmet com CSP
- CORS com whitelist explícita
- Rate limiting por IP (nginx) + por rota (Throttler)
- Swagger restrito a localhost em produção
- Secrets não commitados, .env.example como template
- `SEED_DEV_USER` bloqueado em produção

### Vulnerabilidades identificadas

| # | Vulnerabilidade | Severidade | Esforço fix |
|---|----------------|-----------|-------------|
| 1 | CSRF desabilitado em produção | Alta | 2h |
| 2 | Timing attack na comparação de CSRF token | Média | 15min |
| 3 | Tenant spoofing via header em webhook routes | Alta | 2h |
| 4 | Containers como root | Média | 30min |
| 5 | Sem OCSP stapling no nginx | Baixa | 15min |
| 6 | Sem WAF (ModSecurity/Cloudflare) | Baixa | 2h |
| 7 | `uncaughtException` não encerra processo | Média | 30min |
| 8 | Dev DB usa superuser (mascara bugs de RLS) | Baixa | 1h |

## 8. Riscos Estratégicos

## 9. Conclusão

O projeto está em boa posição para escalar. As bases técnicas são sólidas (RLS, testes, backup, segurança). Os maiores gaps são operacionais (monitoramento, CD) e de manutenibilidade (God Objects). A Fase 1 (segurança + visibilidade) deve ser executada imediatamente pois tem alto impacto com baixo esforço. A Fase 2 (dívida técnica) é necessária antes de adicionar funcionalidades novas ao bot ou ao frontend.
