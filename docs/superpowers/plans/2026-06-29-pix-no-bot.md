# PIX-no-bot Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Provar o fluxo PIX do bot de WhatsApp ponta-a-ponta na máquina local (gerar → pagar no sandbox → webhook confirmar → pedido liberar), com um guarda que torna impossível tocar o token de produção em dev.

**Architecture:** Backend NestJS. O guarda anti-`APP_USR` fecha primeiro e isolado (Task 1). Depois, o checkout do bot passa a coletar os dados de cumprimento antes de criar o pedido/PIX (entrega na Task 2, retirada na Task 5), o fluxo do bot perde os buracos (Task 3), e o caminho `approved → CONFIRMADO` ganha o teste que faltava + a expiração do PIX alinhada ao TTL existente (Task 4).

**Tech Stack:** NestJS 11, TypeORM 0.3, PostgreSQL 15 (RLS via `set_config('app.current_tenant_id')`), jest (integração contra `ucm_test_motor` via túnel SSH `localhost:5544`), MercadoPago SDK (`mercadopago`).

**Spec:** `docs/superpowers/specs/2026-06-29-pix-no-bot-design.md`

## Global Constraints

- **PIX exclusivamente.** Não tocar cartão/boleto (`processCardPayment`, `processBoletoPayment`, `createCardPayment`, `createBoletoPayment`) — nem código nem teste.
- **NUNCA usar o token `APP_USR` fora de produção.** Testes usam token `TEST-...` ou mocks. Nenhum teste pode chamar o MercadoPago real com `APP_USR`.
- **`commitSale` (baixa/VENDA no ledger) é no `PRONTO`** ([orders.service.ts:647-648](../../backend/src/modules/orders/orders.service.ts)). Não mover pro pagamento. Confirmar pagamento (`CONFIRMADO`) **não** baixa nem libera estoque.
- **Expiração do PIX (D5):** reutilizar `ORDER_PAYMENT_TTL_MINUTES` (default 60, lido em [stock-sweeper.service.ts:77](../../backend/src/modules/products/stock-sweeper.service.ts)). **Não inventar** um segundo TTL.
- **`payer.email` = placeholder** (`customer@example.com` já é o default do provider). Não coletar e-mail real.
- Commits em inglês; código e comentários em português; trailer `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>`.
- Testes de integração exigem o túnel SSH pro `ucm_test_motor` de pé (`backend/.env` apontando pra `localhost:5544`).
- **Token de teste local:** o `backend/.env` de desenvolvimento deve usar `MERCADOPAGO_ACCESS_TOKEN=TEST-...` (ou vazio). Com a Task 1 no lugar, um `APP_USR` em dev impede o backend de subir — isso é o comportamento desejado.

- [ ] **Step 3: Implement the guard**

No topo de `mercadopago.provider.ts`, exportar a constante e a checagem:

```typescript
export const PROD_TOKEN_PREFIX = 'APP_USR';
export const PROD_TOKEN_IN_DEV_MSG =
  '[SEGURANCA] Token de PRODUCAO (APP_USR) detectado fora de producao. ' +
  'Use um token TEST- localmente. Operacao bloqueada.';

/** Lança se um token de produção (APP_USR) for usado fora de produção. */
function assertNotProdTokenInDev(token: string): void {
  const isProd = process.env.NODE_ENV === 'production';
  if (!isProd && token.startsWith(PROD_TOKEN_PREFIX)) {
    throw new Error(PROD_TOKEN_IN_DEV_MSG);
  }
}
```

No **construtor**, logo após ler `this.accessToken`, antes de criar o client:

```typescript
this.accessToken = this.configService.get<string>('MERCADOPAGO_ACCESS_TOKEN') || '';
assertNotProdTokenInDev(this.accessToken); // guarda no boot
```

No **início de `createPixPayment`**, antes do `payment.create`:

```typescript
assertNotProdTokenInDev(this.accessToken); // backstop no momento de criar pagamento
```
